## 화면 언어 분석

1. 기술은 속성색보다 **동작의 문법**으로 먼저 구분된다. 물은 이어지는 붓선, 번개는 정지 뒤 직선 도약, 사랑은 사용자에게 묶인 채찍, 음은 박자가 있는 폭발이다.
2. 이동형 효과는 이미지 중앙이 아니라 **선단**이 충돌 위치에 와야 한다. 모든 이동 원화는 로컬 `+X`를 진행 방향으로 제작하고, `water/flame/thunder/butterfly/mist/wind`의 선단 앵커를 셀 폭의 `0.84`로 맞췄다.
3. 한 장을 회전시키는 방식으로는 준비–전개–소멸을 표현할 수 없다. 각 기술을 `anticipation → contact → follow-through → dissipate` 네 프레임으로 제작했다.
4. 광원은 장식이 아니라 타격 위치를 읽게 해야 한다. 움직이는 선단, 충격점, 필살기를 별도 후보로 제출하고 화면 가까운 후보만 셰이더 광원 예산 안에서 선택한다.
5. 피해 판정과 잔상은 분리한다. 안개의 복수 잔상은 거짓 칼날이고 실제 피해선은 하나뿐이며, 번개의 여러 시각 선도 한 시전당 한 번만 피해를 준다.

## 스프라이트와 Three.js 결합

- `technique-atlas.png`: 2048×2048, 8×8 셀, 13테마 × 4프레임(52셀).
- 본체는 normal-alpha, 보조광은 additive로 렌더링하되 두 인스턴스 배치가 모든 테마를 공유한다.
- `aCell` 인스턴스 속성으로 프레임을 고르고, 이동 투사체는 생성/이동/소멸 상태에 맞춰 프레임을 선택한다.
- 곡선 잔상, 선단 스트릭, 안개 베일, 낫바람, 충격 펄스는 스프라이트와 별개의 절차 메시로 투영한다.
- 실제 `PointLight`를 투사체마다 만들지 않는다. 데스크톱은 이동 2·충격 6·필살 2, 모바일은 1·3·1개의 후보만 공유 surface shader에 주입한다.
- 빠른 투사체는 이전 위치와 현재 위치 사이를 캡슐처럼 검사해 프레임 사이를 통과하는 적도 놓치지 않는다.

## 저작권·출처 원칙

공식/팬 문서의 캡처는 분석 자료로만 보고 제품 자산으로 복제하지 않는다. 배포물은 직접 생성한 비트맵, 절차 지오메트리, 셰이더, 입자로 원작의 동작 언어만 재해석한다.
